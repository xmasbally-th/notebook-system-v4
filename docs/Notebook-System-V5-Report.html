บทที่ 1 บทนำ (Introduction)

1. ความเป็นมาและความสำคัญของปัญหา

ในอดีต การบริหารจัดการพัสดุและครุภัณฑ์ โดยเฉพาะอุปกรณ์คอมพิวเตอร์พกพา (Notebook) ของคณะวิทยาการจัดการ มหาวิทยาลัยราชภัฏลำปาง เจ้าหน้าที่ได้ประยุกต์ใช้เทคโนโลยีสารสนเทศเบื้องต้นอย่าง Google Forms ในการบันทึกข้อมูลการยืมและการคืนอุปกรณ์ เพื่อทดแทนการจดบันทึกลงในสมุดแบบเดิม อย่างไรก็ตาม เมื่อปริมาณอุปกรณ์และการใช้งานเพิ่มมากขึ้น พบว่าการใช้ Google Forms มีข้อจำกัดสำคัญหลายประการ เช่น ไม่สามารถแสดงสถานะความพร้อมของอุปกรณ์ได้แบบเรียลไทม์ (Real-time Status) ทำให้ผู้ใช้งานไม่ทราบว่าอุปกรณ์ใดว่างหรือถูกยืมไปแล้วก่อนที่จะกรอกแบบฟอร์ม อีกทั้งเจ้าหน้าที่ยังประสบปัญหาในการติดตามอุปกรณ์ที่เลยกำหนดคืน เนื่องจากระบบเดิมขาดการแจ้งเตือนอัตโนมัติและการเชื่อมโยงข้อมูลที่เป็นระบบฐานข้อมูลเชิงสัมพันธ์อย่างแท้จริง

ด้วยเหตุนี้ ข้าพเจ้าจึงเล็งเห็นความสำคัญในการยกระดับกระบวนการทำงานสู่ระบบดิจิทัลที่สมบูรณ์แบบยิ่งขึ้น จึงได้ดำเนินการพัฒนา Notebook System V5 ซึ่งเป็นระบบเว็บแอปพลิเคชันที่พัฒนาด้วยเทคโนโลยี Next.js 15 และ Supabase เพื่อแก้ปัญหาที่พบในระบบเดิม ระบบใหม่นี้ไม่เพียงแต่ทำหน้าที่บันทึกข้อมูล แต่ยังครอบคลุมถึงการตรวจสอบสถานะอุปกรณ์แบบทันที การจองอุปกรณ์ล่วงหน้า และการแจ้งเตือนผ่าน Discord Webhook ที่จะช่วยให้ทั้งผู้ใช้งานและเจ้าหน้าที่สามารถสื่อสารและติดตามพัสดุได้อย่างแม่นยำ การเปลี่ยนแปลงจากระบบบันทึกข้อมูลทั่วไปสู่ระบบจัดการทรัพยากรแบบครบวงจรนี้ ถือเป็นก้าวสำคัญในการเพิ่มประสิทธิภาพการบริหารจัดการพัสดุให้มีความโปร่งใส ตรวจสอบได้ และลดภาระงานซ้ำซ้อนของเจ้าหน้าที่ได้อย่างมีประสิทธิภาพ

นอกจากนี้ การพัฒนาระบบในเวอร์ชันล่าสุดยังมุ่งเน้นไปที่ความมั่นคงปลอดภัยของข้อมูลผ่านระบบ Row Level Security (RLS) เพื่อป้องกันการเข้าถึงข้อมูลสำคัญโดยมิชอบ และรองรับการขยายตัวของข้อมูลที่มีปริมาณมากในอนาคต การพัฒนาครั้งนี้จึงไม่เพียงแต่เป็นการปรับปรุงเครื่องมือทำงานจากรูปแบบเดิมที่จำกัดอยู่เพียงการจัดเก็บข้อมูลผ่าน Google Forms แต่เป็นการวางรากฐานด้านนวัตกรรมดิจิทัลที่จะช่วยส่งเสริมภาพลักษณ์และการบริหารจัดการเชิงรุกของคณะวิทยาการจัดการ มหาวิทยาลัยราชภัฏลำปาง ให้มีความเป็นมืออาชีพและยั่งยืนตามมาตรฐานการปฏิบัติงานยุคดิจิทัล

2. วัตถุประสงค์ของโครงการ

1.เพื่อพัฒนาระบบบริหารจัดการยืม-คืนอุปกรณ์โน้ตบุ๊คที่รองรับการใช้งานผ่านอินเทอร์เน็ตที่เข้าถึงได้ตลอดเวลา

2.เพื่อเปลี่ยนผ่านจากการใช้ระบบบันทึกข้อมูลแบบทั่วไป (Google Forms) สู่ระบบจัดการทรัพยากรที่รองรับการทำงานแบบเรียลไทม์

3.เพื่อให้ผู้ดูแลระบบ (Admin) มีความสามารถในการดูรายงานสถิติ และการใช้งานอุปกรณ์ในรูปแบบ Dashboard เพื่อง่ายต่อการตัดสินใจบริหารทรัพย์สิน

4.เพื่อสร้างระบบแจ้งเตือนแบบเรียลไทม์ และระบบ Support Chat ภายในแอปพลิเคชัน เพื่อประสานงานระหว่างผู้ใช้งานกับเจ้าหน้าที่ได้ทันท่วงที

3. ขอบเขตของโครงการ

การพัฒนาระบบมีการแบ่งส่วนสิทธิ์ผู้ใช้ออกเป็น 3 ระดับหลัก คือ

  • ผู้ดูแลระบบ (Admin): มีสิทธิ์ระดับสูงสุด สามารถเพิ่ม ลบ แก้ไข ข้อมูลอุปกรณ์ หมวดหมู่ผู้ใช้ ดูแลระบบสถิติ รวมไปถึงการตอบ Support Chat
  • เจ้าหน้าที่ผู้ปฏิบัติงาน (Staff): สามารถอนุมัติหรือปฏิเสธคำขอยืม การตรวจสอบสภาพเครื่องและทำการบันทึกการส่งคืน และรับหน้าเสื่อแจ้งเตือนต่างๆ
  • ผู้ใช้งานทั่วไป (User): มีสิทธิ์เข้าดูแคตตาล็อกอุปกรณ์ ค้นหาตัวเลือก ขอยืม จองคิวล่วงหน้า ยกเลิกการจอง และการตอบแบบประเมินเมื่อคืนเสร็จสิ้น

4 ประโยชน์ที่คาดว่าจะได้รับ

การนำระบบ Notebook System V5 มาใช้งาน คาดว่าจะเพิ่มประสิทธิภาพในการจัดการสินทรัพย์เทคโนโลยีให้กับองค์กรได้เกินกว่า 80% ป้องกันการสูญหายของทรัพย์สิน และสามารถอ้างอิงหลักฐานทางดิจิทัลหรือ Audit Log ได้จากฐานข้อมูลกลางที่มีความปลอดภัยสูง รวมถึงช่วยลดต้นทุนเวลาการทำงานของเจ้าหน้าที่ฝ่ายสนับสนุนทางด้าน IT

บทที่ 2 แนวคิดและทฤษฎีที่เกี่ยวข้อง (Literature Review)

ในการศึกษาและพัฒนาระบบ Notebook System V5 ผู้จัดทำได้ทำการรวบรวมทฤษฎี หลักการ และซอฟต์แวร์เครื่องมือต่างๆ เพื่อนำมาประกอบและออกแบบระบบให้มีความเหมาะสม ทันต่อเทคโนโลยี และง่ายต่อการปรับขยายในอนาคต ดังรายละเอียดต่อไปนี้

2.1 กรอบแนวคิดการประมวลผลและการพัฒนาด้วย Next.js 15+ (App Router)

Next.js เป็น React Framework ที่ถูกสร้างขึ้นมาเพื่อตอบสนองการทำ Server-Side Rendering (SSR) และ Static Site Generation (SSG) ที่ช่วยให้เว็บแอปพลิเคชันมีประสิทธิภาพสูงขึ้น โหลดหน้าเว็บได้เร็ว และเป็นมิตรต่อ SEO โดยในระบบนี้เราเลือกใช้สถาปัตยกรรมแบบ App Router รุ่นใหม่ของ Next.js 15+ ซึ่งทำให้โครงสร้างของเส้นทาง (Routing) ดูแลรักษาง่ายขึ้น ร่วมกับการใช้ Server Actions เพื่อทำ Data Mutation ได้อย่างปลอดภัยโดยไม่ต้องสร้าง API Gateway ภายนอกให้ซับซ้อน

2.2 ระบบจัดการฐานข้อมูล Supabase (PostgreSQL)

Supabase คือ แพลตฟอร์มที่เป็นโอเพ่นซอร์สซึ่งสร้างบนสถาปัตยกรรม PostgreSQL ที่มีความแข็งแกร่ง มีระบบ Authentication ในตัวที่มีความปลอดภัย และรองรับการทำ Row-Level Security (RLS) เพื่อป้องกันและจำกัดสิทธิ์ผู้ใช้ที่จะเข้าถึงข้อมูลต่างๆ ได้อย่างแม่นยำ ด้วยคุณสมบัตินี้ สิทธิ์ระดับ Admin และ User จะเห็นเฉพาะข้อมูลที่ตนมีสิทธิ์ ไม่สามารถล้วงลูกข้อมูลจาก API สาธารณะได้เด็ดขาด และยังเชื่อมรวมกับการทำงานแบบ Real-time subscriptions ในบางฟังก์ชัน

2.3 กรอบและการแสดงผลด้วย TailwindCSS และ Recharts

สำหรับการออกแบบ User Interface ระบบได้ทำงานสอดประสานกับ TailwindCSS ซึ่งเป็น Utility-first CSS framework เพื่อความรวดเร็วในการจัดตำแหน่งและองค์ประกอบบนหน้าจอ (Responsive Web Design) สามารถรองรับการแสดงผลทั้งบน Desktop และ Mobile ส่วนในด้านการนำเสนอข้อมูลหน้า Admin Dashboard ได้ใช้ Recharts ซึ่งเป็น Component ของ React ในการวาดกราฟเส้น กราฟแท่ง ให้สวยงามและง่ายต่อการตีความทางสถิติ

2.4 ทฤษฎีความปลอดภัยและการตรวจสอบโครงสร้างข้อมูลด้วย Zod

Zod คือ Schema Declaration และ Validation library สำหรับ TypeScript ในระบบของ Notebook System V5 มีระบบฟอร์มทั้งการสมัคร บันทึกอุปกรณ์ และขอยืมข้อมูล จึงมีการประยุกต์ใช้ Zod Schema เพื่อตรวจสอบ (Validate) ทุกข้อมูลขาเข้าก่อนที่จะส่งบันทึกลงใน Database เพื่อป้องกันปัญหา XSS หรือ SQL Injection รูปแบบใหม่ๆ

บทที่ 3 การวิเคราะห์ระบบ (System Analysis)

3.1 การวิเคราะห์ระบบงานเดิม

ระบบการยืมและคืนอุปกรณ์โน้ตบุ๊คในอดีต มักใช้การกรอกในแฟ้มเอกสารกระดาษหรือการลงทะเบียนผ่าน Google Forms แบบง่ายๆ ข้อเสียของกระบวนการเหล่านี้คือ 1) ไม่สามารถตรวจเช็คได้ทันทีว่า ณ เวลานั้นอุปกรณ์เครื่องใดกำลังว่าง 2) ข้อผิดพลาดของการส่งคืนล่าช้าไม่ได้รับการติดตามหรือไม่มีการแจ้งเตือนอัตโนมัติ และ 3) ภาระตกอยู่กับผู้ปฏิบัติงาน (Staff) ที่จะต้องตรวจคัดกรองข้อมูลทั้งหมดเอง

3.2 ความต้องการของระบบ (System Requirements)

ผลจากการวิเคราะห์ปัญหา นำไปสู่การวางข้อกำหนดและฟังก์ชันการทำงานใหม่ (Functional Requirements) ดังต่อไปนี้

  1. ระบบต้องให้ผู้ใช้งานเข้าสู่ระบบ (Login) ด้วย Google Account ได้อย่างปลอดภัย
  2. ระบบต้องแสดงสถานะของอุปกรณ์แยกเป็น ว่าง ถูกยืม ซ่อมบำรุง เลิกใช้งาน ได้แบบทันที
  3. ระบบสำหรับ Admin ต้องสามารถจัดการฐานข้อมูลอุปกรณ์ ดูประวัติย้อนหลังได้ไม่จำกัด
  4. ระบบต้องมีช่องทาง Support Chat ภายใน เพื่อสื่อสารระหว่างเจ้าหน้าที่และผู้ใช้งาน
  5. ระบบต้องส่งแจ้งเตือนให้กับผู้ใช้ (ผ่านแอปพลิเคชันและ Discord Webhook) ในทุกๆ State ของธุรกรรม

บทที่ 4 การออกแบบระบบ (System Design)

4.1 สถาปัตยกรรมของระบบ (System Architecture)

ระบบถูกออกแบบบนสถาปัตยกรรมแบบ Client-Server Architecture ในรูปแบบของ Cloud-Native โดยส่วนของเบราว์เซอร์หรือไคลเอนต์ (Client) จะทำการเรนเดอร์เนื้อหาจาก Next.js Web Server โดยอาศัยข้อมูลเชิง Relational ผ่าน Supabase API ในการจัดการธุรกรรมสถาปัตยกรรมโครงสร้างหลักจะประกอบไปด้วย 3 Layers ด้วยกัน คือ 1. Presentation Layer ทำโดย React Components 2. Application Logic ควบคุมโดย Next.js Server Actions และ 3. Data Tier ดูแลโดย PostgreSQL (Supabase) Database

4.2 การออกแบบโครงสร้างฐานข้อมูล (Database Schema)

ตารางข้อมูล (Tables) ถูกร่างขึ้นตามหลักของ Entity-Relationship Model (ERM) เพื่อการเข้าถึงอย่างสัมพันธ์กัน ฐานข้อมูลหลักประกอบด้วยตารางข้อมูลดังต่อไปนี้:

รายละเอียดโครงสร้างตารางข้อมูล

Table Name คำอธิบาย (Description) การเชื่อมโยง (Relationships)
profiles จัดเก็บข้อมูลรายชื่อผู้ใช้ที่ผ่าน Auth จาก Google แล้ว เช่น ชื่อ เบอร์ติดต่อ ตำแหน่งภาควิชาและ Role ลำดับสิทธิ์ pk_id อ้างอิงจาก auth.users ของ Supabase
equipment เก็บพารามิเตอร์ฮาร์ดแวร์ ชื่อ ยี่ห้อ Serial Number และสถานะใช้งาน ณ ปัจจุบัน fk_category_id อ้างไปยัง categories
loan_requests เป็น Core Table หลักสำหรับเก็บบันทึกการออกคำขอยืม การตั้งสถานะ วันที่จะรับและส่งคืน เวลาสร้าง fk_user_id (profiles), fk_equipment_id (equipment)

staff_activity_log ร่องรอยการทำงาน (Audit log) รวบรวบทุกการกระทำของ Staff เพื่อป้องกันการทุจริต fk_staff_id, action_timestamp

4.3 การออกแบบการตรวจสอบความปลอดภัย (Security & Concept Design)

เนื่องจากเป็นระบบสารสนเทศที่มีข้อมูลส่วนบุคคลของผู้อยู่ในภาควิชา การป้องกันการรั่วไหลจึงอาศัยตรรกะ Row Level Security (RLS) แบบเข้มงวดของ Supabase อาทิ:

  • เฉพาะ Admin หรือ Staff เท่านั้นที่สามารถเรียกดูรายการ profiles ทั้งหมดของคนในระบบด้วยกันได้
  • ผู้ใช้งานสาย User จะสามารถ Query คำขอยืม loan_requests ของตัวเองได้เท่านั้น (Where user_id = auth.uid()) ไม่สามารถดึงของเพื่อดูของเพื่อนได้
  • ปิดกั้นการลบข้อมูล (Delete Restriction) ทุกกรณีในประวัติการยืม หากไม่ต้องการให้แสดงให้ทำสถานะซ่อน (Archived) แทน เพื่อรักษาความสมบูรณ์เชิงสถิติ

บทที่ 5 การพัฒนาและการทดสอบระบบ (System Development & Testing)

5.1 การพัฒนาระบบ (System Development)

ในขั้นตอนการพัฒนา โปรแกรมถูกเขียนขึ้นด้วยภาษา TypeScript โดยรันสคริปต์อยู่บน Node.js Environment การแบ่งแฟ้มหลักตามโครงสร้างของ Next.js ตัวระบบรวมมีโฟลเดอร์ปฏิบัติการหลักคือ โฟลเดอร์ app นอกเหนือจากนี้ ในระดับการติดต่อเพื่อ Query ฐานข้อมูล ได้มีการสร้างโฟลเดอร์ lib/data/ เพื่อจำกัดความซับซ้อนและการตรวจสอบ Validation จาก Zod Schema ในโฟลเดอร์ lib/schemas/

ตัวอย่างการประยุกต์ใช้เครื่องมือพัฒนา

ระบบแจ้งเตือนได้รับการพัฒนาให้สามารถยิง Trigger ไปยังห้องสนทนาของ Discord Server กลางหน่วยงาน โดยนำ Endpoint Webhook มาฝังผ่าน lib/actions/discord.ts เมื่อผู้ใช้ส่งคำขอ หรือเมื่อ Admin อนุมัติสำเร็จ ระบบจะส่ง Payload ในรูปแบบของ Embed Rich Text Card ทันที ซึ่งสามารถปรับแต่ง Dynamic ได้ผ่าน system_config table ใน Database

5.2.1 การทดสอบระดับโมดูล (Unit Testing และ Integration)

ได้ทำการทดสอบฟังก์ชันหน้าบ้าน อาทิ การกดเปิดหน้าจอป๊อบอัป การแสดงผลคำขอยืม การฟอร์แมตวันที่และการคำนวณวันเกินกำหนด (Overdue Calculator) รวมถึงการทดสอบ Server Actions ว่าจะสามารถคืนค่าพารามิเตอร์ Error อย่างถูกต้องไปยังฝั่ง Client Component หากส่งวันที่ล้าหลังเข้าระบบ หรือรูปแบบหมายเลขลำดับไม่ถูกต้อง เป็นการป้องกันในระดับแรก

5.2.2 การทดสอบยอมรับจากผู้ใช้ (User Acceptance Testing - UAT)

ในการจัดทำโครงการ ได้ให้กลุ่มตัวอย่างที่เป็นนักศึกษา อาจารย์และผู้รับผิดชอบคลังพัสดุ เข้ามาทดลองใช้ระบบในสภาวะแวดล้อมเสมือนจริง (Staging Database) จากผลการทดสอบพบว่า

  • หน้าตาเว็บไซต์สามารถนำเสนอข้อมูลได้อย่างครบถ้วนและตอบคำถามที่ผู้ใช้อยากรู้ (ความจุ RAM, ตำแหน่งจัดเก็บ)
  • ผู้ใช้งานกลุ่ม Staff ชื่นชมระบบ Dashboard และกราฟสรุปจำนวนรอบการยืมที่แสดงสถานะเป็นก้อนสีต่างๆ ชัดเจนในหน้า Admin/Loans
  • พบข้อสังเกตเรื่องความสับสนเรื่องสถานะคำขอในครั้งแรก ซึ่งผู้พัฒนาได้แก้ไขโดยการเพิ่ม Badge สถานะสีสดใสกำกับพร้อมความหมายแนบไว้ทุกหน้าคำขอของฉัน

บทที่ 6 การติดตั้งและการใช้งาน (System Implementation)

6.1 การตั้งค่าระบบและการติดตั้ง (Deployment)

การติดตั้งแอปพลิเคชันสู่เครือข่ายอินเทอร์เน็ตสาธารณะ (Production) ใช้ระบบคลาวด์แพลตฟอร์ม Vercel ซึ่งรองรับสถาปัตยกรรมของ Next.js อย่างไร้รอยต่อ โดยเชื่อมต่อร่วมกับ GitHub Repository ด้วยกระบวนการ CI/CD (Continuous Integration / Continuous Deployment) ทำให้การเขียนโค้ดเวอร์ชันล่าสุดอัปเดตขึ้นสู่โฮสติ้งทุกครั้งที่มีการเปลี่ยนข้อมูลบน Branch main ส่วนทางด้านระบบเก็บรักษาข้อมูล (Database) ใช้โครงสร้างของระบบคลาวด์บน Supabase Platform (PostgreSQL AS-A-Service)

ตัวแปรสภาพแวดล้อม (Environment Variables) ที่จำเป็น:

  • NEXT_PUBLIC_SUPABASE_URL : สำหรับเชื่อมต่อกับโปรเจกต์ฐานข้อมูล
  • NEXT_PUBLIC_SUPABASE_ANON_KEY : กุญแจการเข้าถึงสำหรับ Client
  • SUPABASE_SERVICE_ROLE_KEY : กุญแจแอดมินใช้เฉพาะฝั่งเซิร์ฟเวอร์เท่านั้น ห้ามเผยแพร่สู่เบราว์เซอร์

6.2 คู่มือการใช้งานสำหรับผู้ใช้ทั่วไป (User Guide Snippet)

1. ไปที่ URL เว็บไซต์เข้าจากอุปกรณ์ใดก็ได้ คลิกเครื่องมือ "เข้าสู่ระบบด้วย Google"

2. การระบุข้อมูลบัญชีครั้งแรกให้ครบ ต้องรอ Admin 1 วันทำการเพื่อเปิดสถานะบัญชีอนุมัติ

3. เลือกไปยังหน้าจอ อุปกรณ์ พิมพ์หากรรไกรหา โน้ตบุ๊ค ฮาร์ดแวร์ เมื่อเจอสิ่งที่ต้องการ ให้คลิกไปที่การ์ด กดเลือกปุ่มยืนยันส่งขอยืมในแบบฟอร์ม

4. รอการตรวจสอบ (สถานะ รอการอนุมัติ สีเหลือง) หากสำเร็จจะเป็นสีเขียว ให้นำหลักฐานรหัสคิวอาร์โค้ดไปแสดง ณ ศูนย์อุปกรณ์เพื่อรับของกลับบ้าน และโปรดเคารพกฎระยะเวลาส่งคืน

6.3 การสนับสนุนและการแชทสนทนาระหว่างผู้ดูแล

หากมีปัญหาเฉพาะกรณีระหว่างบุคคล สามารถกดที่ไอคอนจุดแชท ที่มุมล่างขวา 💬 เพื่อตั้งคำถามถึงผู้ดูแลพัสดุโดยตรงได้ทันที เสมือนโปรแกรม Messenger ช่วยลดระยะเวลาที่ต้องโทรหาหรือเดินมาติดต่อด้วยตนเองที่ห้อง

7.1 สรุปผลการพัฒนาระบบ

โครงการ Notebook System V5 ประสบความสำเร็จตามวัตถุประสงค์ที่ตั้งไว้ ระบบสามารถช่วยการขอยืม จอง และคืนอุปกรณ์ได้โดยไม่ใช้ระบบกระดาษ ผสานความสะดวกของ Web Application ทั้งในระบบติดตามสถานะและการสื่อสารข้อซักถามผ่านระบบ Support Chat โดยจากการวัดผลระหว่างการออกแบบและนำไปประยุกต์ใช้จริง พบว่าระบบได้ลดภาระกระบวนการทำงานซ้ำซ้อนของเจ้าหน้าที่คลังลงได้อย่างเป็นรูปธรรม และยังคงได้รับสถิติและแบบรายงานอย่างทรงพลังผ่านพอร์ทัลสิทธิ Admin ได้อย่างถูกต้องและแม่นยำ ทุกส่วนบนหลักทำงานบนระบบความปลอดภัยเชิงโครงสร้างของ Cloud Solutions ที่ไม่ต้องคอยกังวลเรื่องการสำรองข้อมูลรายวัน

7.2 อภิปรายผล

สืบเนื่องมาจากโครงการนี้ใช้ Framework ระดับโลกอย่าง Next.js ในการก่อร่างโครงสร้าง ทำให้ความลื่นไหลของหน้าเว็บไซต์ และ TTFB (Time To First Byte) ต่ำมากๆ ทำให้เกิดขีดความสามารถการให้บริการผู้พร้อมกันในปริมาณสูงในช่วงสอบของมหาวิทยาลัยโดยไม่เกิดข้อผิดพลาด นอกจากนั้นการเลือกใช้ Recharts ในโมดูลของ Dashboard ให้ผลเชิงสถิติที่สวยงาม สามารถดึงดูดผู้ประเมินผลบริหารได้เป็นอย่างดี อย่างไรก็ตามในการตั้งค่าชั้นสูงของ Supabase RLS Policy นั้นเคยเกิดปัญหา Bug ที่ผู้ใช้ทุกคนเข้าไม่ถึงข้อมูลของตัวเองใน Version ก่อนหน้า ซึ่งก็ได้รับการแพตซ์และแก้ไขประวัติเป็นระบบ RLS Hardening ครบถ้วนแล้วใน V5 ฉบับนี้

2. การนำระบบพยากรณ์และปัญญาประดิษฐ์ (AI) เข้ามาใช้ในการวิเคราะห์พฤติกรรมการยืม และพยากรณ์อายุชิ้นส่วนซ่อมบำรุงว่าเมื่อไหร่เสื่อมสภาพที่จะต้องปลดเพื่อจัดสรรงบประมาณล่วงหน้า

3. เพิ่มตัวดึงข้อมูลด้วยปฏิทินยืมอุปกรณ์ (Calendar Dashboard view) เพื่อให้เห็นกราฟเส้นยาวยืมคืนชัดเจนเสมือนระบบโรงแรมมากยิ่งขึ้น

[ภาพแสดงหน้าจอฟอร์มการกดยืมและปฏิทิน จำลอง กรุณาแนบภาพประกอบจริงภายหลังจากการใช้งานระบบ]
